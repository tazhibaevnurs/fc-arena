## 2. Узкие места (Bottlenecks)

На основе раздела **«Что не готово / ограничено»** из PROJECT_REPORT.md и кода.

### Критические препятствия

1. **~~Нет автоматических тестов~~** ✅ **Закрыто**  
   В `accounts/tests.py` добавлены тесты: активация подписки по Robokassa Result (в т.ч. отклонение при неверной подписи), активация по 2Checkout IPN, show_ads = False для PRO, аноним/Free видят рекламу.

2. **Реклама не приносит доход до подстановки реальных кодов**  
   В шаблонах и `ads_manager` используются заглушки. Без настроенных AdSense/РСЯ (ca-pub-..., R-A-..., слоты в админке) $5k/мес с рекламы недостижимы.

3. **~~Нет аналитики и метрик конверсии~~** ✅ **Закрыто**  
   В base.html подключены GA4 и Яндекс.Метрика (при `GA4_MEASUREMENT_ID` и `YANDEX_METRIKA_ID` в .env). События: subscription_page_view, checkout_click (plan, provider), payment_success, payment_fail.

4. **~~Кэш по умолчанию — LocMem~~** ✅ **Закрыто**  
   При заданной переменной `REDIS_URL` в settings используется `RedisCache` (Django 4.0+). Без REDIS_URL — LocMem для разработки.

5. **Нет мониторинга и алертов**  
   Падение IPN/Result URL, рост 5xx или таймауты не отслеживаются. Возможны «тихие» сбои активации подписки и потеря доверия пользователей.

6. **Статика и медиа**  
   Нет своей папки `static/` в репозитории, только настройки. Риск битых стилей/скриптов после деплоя, если не настроен `collectstatic` и раздача статики.

7. **Документация и онбординг**  
   Нет README с установкой и деплоем. Усложняет онбординг разработчиков и операционные процедуры при инцидентах.

### Рекомендации для устойчивости при росте трафика

- **Платежи:** логировать все вызовы IPN/Result (id, статус, ответ) в БД или в файл; при ошибке — retry и алерт; добавить тесты на проверку подписи и активации подписки.
- **Кэш:** в production использовать Redis (или аналог), один бэкенд для всех воркеров; вынести сессии в кэш при необходимости.
- **Нагрузка:** добавить rate limiting на API (generate-teams, team-suggestions) и на логин/регистрацию; при росте — CDN для статики, опционально кэш страниц (например, публичных m/, t/).
- **Наблюдаемость:** интеграция с системой мониторинга (логи, метрики, алерты при 5xx и сбоях платежей).

## 4. Прогноз: чего не хватает в User Journey для конверсии в $6

При текущем функционале пользователь может играть турниры бесплатно (один профиль, базовая генерация команд, без истории и без публичных ссылок). Чтобы он **захотел** заплатить $6/мес, в пути пользователя не хватает следующего.

### 4.1. Ощутимая боль бесплатной версии

- **Сейчас:** ограничения PRO описаны текстом (нет истории, нет публичных ссылок, реклама).  
- **Не хватает:** чтобы пользователь **регулярно упирался** в эти ограничения в продукте, а не только на странице тарифов.  
  **Рекомендации:**  
  - После завершения турнира показывать блок: «Сохранить турнир в историю и поделиться результатом? Доступно в PRO» с кнопкой «Перейти на PRO».  
  - При попытке создать второй игровой профиль или хайлайт — не только редирект на подписку, но и короткое модальное окно: «Несколько профилей и публичные ссылки — в PRO. $6/мес».

### 4.2. Ценность PRO до оплаты

- **Сейчас:** ценность PRO видна в сравнении на странице подписки.  
- **Не хватает:** «примерить» выгоду до оплаты.  
  **Рекомендации:**  
  - Показать превью «Как будет выглядеть твоя публичная страница победы» (скрин или шаблон с подставленным ником/турниром) по кнопке «Пример публичной ссылки» на странице подписки или после финала турнира.  
  - Один бесплатный хайлайт на аккаунт (например, только для последнего турнира) с водяным знаком «PRO» или ограничением по времени — чтобы дать попробовать шеринг, а затем предложить PRO для без ограничений.

### 4.3. Социальное доказательство и срочность

- **Сейчас:** нет отзывов, счётчиков, ограничений по времени.  
- **Не хватает:** доверия и лёгкого импульса «купить сейчас».  
  **Рекомендации:**  
  - На странице подписки: «Уже N организаторов турниров используют PRO» (если есть данные) или отзыв/кейс.  
  - Ограниченное по времени предложение: «Первый месяц за $3» или «Год со скидкой 30% до конца месяца» (если допустимо по платёжкам и бухгалтерии).

### 4.4. Упрощение решения и снижение трения

- **Сейчас:** нужно залогиниться, выбрать план и платёжку, уйти на внешний сайт.  
- **Не хватает:** минимизации сомнений и шагов.  
  **Рекомендации:**  
  - На главной и после создания турнира — заметный, но не навязчивый CTA: «Без рекламы и с историей турниров — PRO $6/мес».  
  - Сохранять выбранный план (месяц/год) в session и при возврате с платёжки с ошибкой показывать тот же план с кнопкой «Повторить оплату» и кратким объяснением («Оплата не прошла. Можно попробовать снова или другой способ»).

### 4.5. Повторное вовлечение после регистрации

- **Сейчас:** после регистрации пользователь попадает на главную; напоминания о PRO только при явном заходе в «Тарифы».  
- **Не хватает:** напоминания о ценности PRO в момент, когда она максимально релевантна.  
  **Рекомендации:**  
  - После 1–2 проведённых турниров (или при первом завершении турнира) показывать сообщение/баннер: «Ты уже провёл N турниров. С PRO сохранишь историю и сможешь делиться ссылкой на результат.» + кнопка «Узнать про PRO».  
  - Опционально: разовая email-рассылка через 1–2 дня после первого турнира с предложением PRO и ссылкой на страницу подписки.
